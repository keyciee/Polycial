Planning

14-Day Sprint Plan

A structured two-week development cycle following agile principles — two phases per week with clear ownership and deliverables.

Days Phase Deliverables Owner
1–2 Setup Project Foundations GitHub repo created, branching strategy agreed, README skeleton, tech stack confirmed, game concept locked in Both
3–4 Build Core Game Loop Board layout rendered, dice rolling mechanic, token movement, basic property landing logic, turn structure Dev 1
4–5 Build Website Foundation GitHub Pages repo enabled, index.html scaffold, design system (fonts, colours, spacing) defined Dev 2
5–7 Build Game Mechanics Buy/decline properties, rent calculation, houses & hotels, mortgage/unmortgage, Chance/Community decks, Jail rules Dev 1
6–8 Build Website Content Landing page complete with hero, screenshots gallery, how-to-play steps, controls reference, features section Dev 2
8–10 Enhance Advanced Features AI opponents (3 tiers), trading system with counter-offers, political events, bankruptcy auctions, pass-and-play Dev 1
9–10 Enhance Team Page & Docs Team page built, README.md finalised with screenshots, repo structure documented, GitHub Actions workflow Dev 2
11–12 Polish Sound, Timer & UX Web Audio sound effects, turn timer, net worth chart, board hover highlights, save/resume system, stats popup Both
13 Polish Testing & Bug Fixes Full play-through tests (2, 3, 4 player), edge case fixes, AI balance pass, mobile responsiveness check Both
14 Launch Deployment & Submission Final commit, GitHub Pages live deployment verified, README reviewed, all links tested, submission package prepared Both
Technology

Tech Stack

Built entirely with core web technologies — no frameworks, no bundlers, no runtime dependencies.

🌐
HTML5

Single self-contained HTML file for the game. Semantic markup for the website pages. No build step required.

🎨
CSS3

Custom properties, CSS Grid, Flexbox, keyframe animations, backdrop-filter. No CSS framework used.

⚡
Vanilla JS

Pure JavaScript ES6+. DOM manipulation, localStorage API, Web Audio API, getBoundingClientRect for token sprites.

🔊
Web Audio API

Procedurally synthesised sound effects — no audio files. Oscillators with gain envelopes for every game event.

💾
localStorage

Full game state serialised to JSON. Auto-save every 3 turns plus manual save. Resume from any point.

🐙
GitHub Pages

Static site hosting directly from the repository. Zero configuration — push to main and the site deploys automatically.

🔤
Google Fonts

DM Serif Display (headings), DM Sans (body), Space Mono (UI labels). Loaded via CDN link.

🎭
Playwright

Used during development to capture live screenshots of every game screen for documentation and the README.

Repository

Repo Structure & Deployment

The entire project lives in a single GitHub repository, deployed to GitHub Pages with one click.

  polycial / (repository root) 
polycial/
├── index.html        # Game landing page (GitHub Pages home)
├── team.html         # Team page with roles & GitHub links
├── polycial.html     # The complete game (single file)
├── README.md         # Project documentation
├── ss_start.png      # Screenshot: start screen
├── ss_game.png       # Screenshot: game board
├── ss_midgame.png    # Screenshot: mid-game state
├── ss_modal.png      # Screenshot: property modal
├── ss_settings.png   # Screenshot: settings panel
└── ss_stats.png      # Screenshot: stats popup
01
Fork or Create Repo
gh repo create polycial --public

Create a new public repository on GitHub. Name it anything — the Pages URL will be username.github.io/repo-name.

02
Push All Files
git add . && git commit -m "Initial release" && git push

Push the game file, both HTML pages, README, and all screenshot images to the main branch.

03
Enable GitHub Pages
Settings → Pages → Branch: main → / (root)

In your GitHub repository settings, enable Pages from the main branch root. No build process needed.

04
Site Is Live
https://[user].github.io/polycial/

Within 60 seconds your site is live. Every push to main auto-deploys. Update the nav GitHub link in both HTML files.
